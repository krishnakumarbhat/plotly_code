Project Architecture & Design Notes
1. High-Level Architecture: The "Manager-Worker" Model
Since you are operating in an HPC (High-Performance Computing) environment with Slurm, you cannot treat the cluster like a single server. You must separate the Control Plane (Web App) from the Compute Plane (Heavy Tools).

Node A: Login/Head Node (The "Manager")

Role: User Interface & Job Orchestration.

Software: Flask Web Application, PostgreSQL Database.

Resource Usage: Low CPU, Low RAM. Runs 24/7.

Action: When a user clicks "Run", this node does not run the code. It generates a Slurm script (sbatch) and submits it to the queue.

Node B: Compute/GPU Nodes (The "Workers")

Role: Heavy Execution (KPI, IPS, DHTML, LLM).

Software: Singularity Containers (.sif files) containing heavy libraries (PyTorch, TensorFlow, etc.).

Resource Usage: High CPU, GPU utilization. Runs only when requested.

2. Technology Stack
Backend Framework: Flask (Python).

Container Engine: Singularity/Apptainer (Required for HPC clusters). You will build Docker images locally and convert them to .sif files on the cluster.

Database 1 (Relational): PostgreSQL.

Purpose: User management (NetID/Password), Session history, Job logs (Status: Queued, Running, Failed, Success).

Database 2 (Vector): ChromaDB.

Purpose: Storing embeddings of tool outputs so the Local LLM can "remember" context.

Job Scheduler: Slurm (using sbatch for background tasks and srun for interactive tasks).

LLM: Qwen (or similar) running as a persistent service on a GPU node.

3. Critical Workflows
A. The Job Submission Workflow (KPI/IPS/DHTML)

User: Uploads a file or selects a path like /mnt/cluster/data/input.h5 via the Flask UI.

Flask:

Creates a unique job_id.

Inserts a record into PostgreSQL: (job_id, user, status="QUEUED").

Constructs a shell command: sbatch --parsable --output=log_%j.txt run_kpi.sh <input_path>

Executes command using Python subprocess.

Slurm: Allocates a Compute Node and runs the Singularity container.

Worker (Compute Node):

Runs the heavy algorithm.

Writes output files to disk.

(Optional) Calls back to a Flask API endpoint to update status to "COMPLETED".

B. The "Chat with AI" Workflow

Challenge: You cannot sbatch every chat message (too slow).

Solution: Host the LLM as a "server" inside the cluster.

Start one Slurm job that requests a GPU for 24 hours (or indefinitely).

This job runs Qwen and exposes an API (e.g., http://10.0.0.5:8000/generate).

Your Flask app sends user prompts to 10.0.0.5:8000 and displays the response.

4. File System Strategy
Local Laptop Uploads: Flask saves these to a "scratch" directory that is mounted and visible to the compute nodes (e.g., /scratch/user/uploads).

Cluster Paths: The UI must include a file browser component that allows users to click-select files from /mnt/cluster to avoid typo errors and security risks.

Important env / DB notes

DATABASE_URL controls DB backend:
sqlite: sqlite:////absolute/path/to/dbfile
postgres: postgresql://user:pass@host:5432/dbname
If DATABASE_URL is unset on Linux, the app may try Postgres at localhost:5432 вЂ” set DATABASE_URL to use SQLite or point to a running Postgres.
.env contains default Postgres URL; change it for dev or cluster.
Common issues & fixes

Jinja2 / MarkupSafe import error ("cannot import name soft_unicode"):
Fix: pin MarkupSafe to 2.0.1 in requirements.txt and requirements-min.txt and rebuild container.
psycopg2 OperationalError (Connection refused):
Means Postgres unavailable. Use SQLite for dev or start Postgres (Docker or system) and set DATABASE_URL accordingly.
all_services.sif vs all_services.simg:
Extension does not matter. You can cp or mv .sif -> .simg. Scripts can be updated to normalize.
WSL networking:
Container binds 0.0.0.0:5001; access from Windows at http://localhost:5001. If blocked, use wsl --shutdown and restart or set portproxy in Windows.
File ownership/permissions after build:
apptainer build might create root-owned image; chown to your user for easy scp: sudo chown "$USER:$USER" all_services.simg
