🔍 Why is Southfield slow in real use?

When you are logged in, loading the main page /html triggers a bottleneck of sub-tasks that do not occur on /html/kpi:

📦
1. SQLite DB on Network Mount Lock-Contention

The SQLite database hpc_tools_dev.db is located in the cache directory. On Southfield, projects are hosted on an SMB/NFS cluster mount (/mnt/usmidet). SQLite does heavy file-locking; network locking on SMB/NFS triggers remote protocol calls that waste 100-300ms per lock write/read. Southfield's locking IO overhead slows down every database transaction.

🔄
2. Sequential Job Sync Loop on /html

When loading /html, the dashboard queries 10 recent jobs and executes _sync_runtime_job() in a loop. For each job, it reads parameters, queries SQLite, and fetches status. On /mnt/usmidet network storage, this sequential SQLite lock sequence delays response time by 2.5s - 5.0 seconds! On the other hand, /html/kpi avoids the 10-job sync loop completely!

🔌
3. Blocking Broker Sockets & Firewall Drops

Before loading the dashboard, the server runs broker_client.ping() with a 60.0 seconds connection timeout. If the background ZMQ broker process on Southfield (hpcc_main.py) is dead or firewalled with packet drops, the Flask thread blocks and hangs until the timeout expires or eventually returns an error!

⚡ Why is Krakow fast?

Krakow's environment is highly integrated and benefits from localized hardware and file storage systems:

🚀
1. Fast Lustre/GPFS Cluster Directory locking

Krakow hosts projects in /net/8k3/..., which is typically a high-performance Lustre cluster or has local SSD cached disks for SQLite. SQLite database block operations do not block on file-locks like they do in Southfield's SMB mounts.

💚
2. Responsive Background Broker Daemon

The Krakow backend has its broker daemon hpcc_main.py running actively and listening on the server. The connection completes immediately and broker_client.ping() is answered in < 2 milliseconds.

🪵
3. Local RAM Heartbeats for Gunicorn

Krakow system provides /dev/shm (Shared Memory disk) which Gunicorn uses for its heartbeats. If Southfield lacks write permission or has no /dev/shm, Gunicorn uses file-based tracking on NFS, constantly blocking workers.
